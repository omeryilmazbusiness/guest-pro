PHASE 2 — DESIGN THE MOST SECURE QR AUTO-LOGIN FLOW
==================================================

Implement a production-grade QR login architecture.

SECURITY GOAL:
The QR code must not simply expose a raw reusable guest key if a more secure design is possible.

PREFERRED ARCHITECTURE:
- when guest is created, generate a secure auto-login token or magic-link token
- the QR code should contain a URL such as:
  /guest/auto-login?token=...
- the token should be:
  - signed or securely random
  - short-lived and/or single-use
  - mapped to the correct guest
  - validated server-side
- after successful use:
  - create the guest session securely
  - redirect to guest dashboard
  - revoke or invalidate token if using single-use strategy

SECURITY REQUIREMENTS:
- choose the strongest practical design for this app
- prefer:
  - short-lived token
  - single-use token
  - hashed token storage in DB
  - expiry checks
  - audit logging
- do NOT store sensitive reusable secrets in QR if avoidable
- do NOT trust client-only checks

You may choose a strong pattern such as:
- guest auto-login token table
- hashed token stored server-side
- token family / issuance record
- revoke after first successful login
- explicit expiry time
- optional reissue/refresh flow
