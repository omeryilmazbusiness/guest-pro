You are a senior full-stack software architect, product analytics designer, AI insight workflow designer, and production-grade engineer.

Your task is to inspect, redesign, and harden the “Quick Report” feature inside the existing “Guest Pro” manager dashboard so it becomes a production-ready, manager-focused operational intelligence tool.

IMPORTANT EXECUTION RULES:
- First inspect the existing codebase, manager dashboard, request/task model, request status lifecycle, timestamps, room data, guest data, AI integration, analytics/reporting structure, and UI architecture.
- Do NOT rebuild the app from scratch if working systems already exist.
- Preserve working auth, manager/staff dashboards, request creation flows, status updates, route protection, and stable APIs.
- Refactor only where there is real architectural, maintainability, analytics, or UX value.
- The final result must be mobile-friendly, premium, compact, operationally useful, SOLID, layered, and production-ready.

PROJECT CONTEXT:
Guest Pro already has:
- guest requests
- request types such as food/support/care
- request statuses such as Açık / İşlemde / Tamamlandı
- room numbers
- guest information
- manager dashboard
- AI integration

We now need to refine the “Quick Report” feature.

PRIMARY GOALS:
1. When the manager clicks “Quick Report”, the system must analyze the CURRENT requests and generate a compact operational report
2. The AI summary and recommendation must focus approximately 80% on:
   - requests
   - complaints
   - repeated problems
   - unresolved/friction-heavy issues
3. The report must analyze:
   - what issue each guest has
   - in which room it happened
   - what type of request/complaint it is
   - whether it is still open, in progress, or completed
   - how long it waited
   - how long it took to resolve
4. The report must produce:
   - a short summary
   - a short problem-focused analysis
   - short manager-oriented recommendations
5. Recommendations must focus on:
   - reducing manager workload
   - improving service quality
   - reducing repeated complaints
   - improving response speed
6. Keep the feature compact, premium, and highly operational
7. Build it with a clean SOLID architecture

PHASE 2 — REDEFINE QUICK REPORT PURPOSE
==================================================

The Quick Report is NOT a generic dashboard summary.

It must be a manager-focused operational intelligence popup/report with strong focus on complaints and service requests.

Approximately 80% of the output should focus on:
- active requests
- repeated complaints
- unresolved issues
- delays
- friction points
- service failures or recurring support patterns

Only the remaining smaller portion should focus on broader operational context.

The output must feel like:
- “What is going wrong right now?”
- “Where are the repeated problems?”
- “Which rooms/guests need attention?”
- “How can we reduce repeated problems and manager workload?”

==================================================

The analysis must prioritize useful operational interpretation.

AI should detect patterns such as:
- repeated Wi-Fi complaints
- repeated extra pillow requests
- repeated minibar refresh requests
- repeated cleaning requests
- rooms with unusually frequent support tickets
- requests staying open too long
- bottlenecks between Açık and İşlemde

The report should explicitly connect:
- guest / room / issue / timing / repeated pattern

- “Room 304 and 306 generated repeated Wi-Fi-related support requests.”
- “Extra pillow requests are frequent and mostly come in the evening.”
- “Cleaning requests are taking longer than food requests.”
- “Three support requests remained open for more than X minutes.”

Keep insights very short and operational.
